# 视图

视图是 MySQL从 5.0.1 版本提供的新功能，是一种虚拟存在的表，视图本身并不保存任何数据，本质上对应一条 SELECT 语句。只保存了 SQL 逻辑，不保存查询结果，在使用视图时会动态生成数据。

# 应用场景

  1. 查询结果使用 SQL 较为复杂。

  2. 复用 SQL。

  3. 封装复杂 SQL 的实现细节，简化查询操作。

  4. 保护数据查询安全性。

    某银行为了提高数据安全性，不让私有化产品直接访问它们的生产数据库，而是将需要的查询结果创建对应的视图，再创建一个只拥有视图权限的账号，很好的保证了生产环境数据的安全性。

# 视图注意点

  1. 视图必须唯一命名。（不能跟别的视图和表重名）
  2. 对于可以创建的视图数量没有限制。
  3. 视图可以嵌套，即可以利用从其他视图中检索数据的查询来构造一个新的视图。
  4. ORDER BY 可以用在视图中，但如果从该视图检索数据的 SELECT 中也含有 ORDER BY ，那么视图中的 ORDER BY 会被覆盖。
  5. 视图不能索引，也不能有关联的触发器或默认值。
  6. 视图可以和表一起使用。
  7. 视图一般用于检索（SELECT）而不用于更新（INSERT, UPDATE, DELETE），因为更新一个视图相当于更新其基表，如果不能正确地确定被更新的基数据，则不允许更新。

# MySQL逻辑架构

MySQL 采用的是 C/S 架构，分为服务端和客户端，我们通过客户端向服务端发送 SQL 等来操作数据库。

# 逻辑架构图

# 四层架构

下图是 MySQL 的整体逻辑架构图，按照功能可以划分为四层：连接层、服务层、引擎层和存储层。

# 连接层

客户端首先访问的就是连接层，连接层的作用就是创建客户端和MySQL服务端之间的连接，还包含着对客户端的权限校验。

  • 建立客户端和服务端的连接。
  • 对客户端进行权限校验。

连接池就存在于连接层，连接池的作用是为了方便管理客户端和 MySQL服务端之间的连接。

# 服务层

服务层包含查询缓存、SQL接口、解析器、查询优化器、执行器。

  1. 查询缓存

    当与连接器建立连接之后，若执行的是查询（SELECT）语句，那么连接器会先向查询缓存发起查询，若查询缓存包含数据，则判断数据权限，符合权限后直接返回结果。

    select SQL_CACHE * from class where id =1
    1

    查询缓存的原理

    把查询语句当作 key，查询结果当作 value，使用 K-V 结构存入缓存中。 当数据表的数据发生变化时，要对查询缓存进行更新，不然查询缓存不能成功。

    MySQL8.0官方删除了查询缓存的功能。

  2. SQL接口

    SQL语句的入口，接受用户的SQL命令，支持接受 DMS，DDL语句，还支持视图、存储过程中，并将最终结果返回给用户。

  3. 解析器 - Parser

    解析 SQL 语句，并验证 SQL 语句的合法性。

    • 词法分析，提取 SQL 语句中的关键字，比如 SELECT。
    • 语法分析，检验输入 SQL 的合法性。

  4. 查询优化器 - Optimizer

    针对查询的优化器，SQL 查询语句在执行之前会被查询优化器进行优化。

    比如优化器会根据查询条件来判断走不走索引，连接查询时会决定先查哪个表。

  5. 执行器

    SQL 语句在被优化器优化后，交给执行器执行。执行器在开始执行钱，会判断用户是否拥有具体数据表的操作权限。

    执行器通过调用存储引擎的 API 来执行 SQL 语句。存储引擎负责执行具体的 SQL，并返回执行结果给执行器。执行器获得执行结果后，返回给客户端，若是 SELECT 语句还要将执行结果写入到查询缓存中。

# 引擎层

引擎层就是存储引擎，MySQL 有多种存储引擎，并且设置成了可插拔形式的。不同的存储引擎作用不同。

# 存储层

数据存储层，将数据存储在文件系统中，并完成与存储引擎的交互。
